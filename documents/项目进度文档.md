# ACTrade 项目结构与系统设计文档

## 1. 文档说明

本文档用于说明 ACTrade 项目的当前目录结构、后端分层设计、技术栈、核心数据模型，以及围绕“激活码 / 激活工具”交付能力的扩展设计，供团队内部评审、开发协作与后续归档使用。

---

## 2. 项目概览

ACTrade 是一个交易平台项目，当前以后端 API 工程为核心，基于 Spring Boot 3 + Java 21 开发。  
现阶段已完成用户、商家、商品、商品分类、商品详情等基础能力，并已规划激活码库存和激活工具执行两类交付模式。

## 3. 后端架构设计

### 3.1 分层结构

后端代码采用按业务模块组织、模块内分层的方式：

- `api`：控制器层，对外暴露 REST API
- `api/request`：请求 DTO
- `api/response`：响应 DTO
- `application`：应用服务层，负责业务流程编排
- `domain`：领域模型层，包含实体、枚举等核心对象
- `persistence`：持久化层，包含 Repository 与查询规格
- `common`：公共响应模型、通用异常
- `config`：系统配置
- `web`：Web 层统一异常处理

### 3.2 当前业务模块

- `app_users`：用户模块
- `merchant`：商家模块
- `product`：商品模块

## 6. 核心数据模型设计

### 6.1 基础业务模型

当前系统核心数据模型包括：

- 用户 `app_users`
- 商家 `merchant`
- 商品分类 `product_category`
- 商品 `product`
- 商品详情 `product_detail`

上述表构成当前平台基础商品管理能力。

### 6.2 交付能力扩展模型

为支持不同商品交付方式，系统扩展以下数据实体：

- `activation_tool`：激活工具主表
- `activation_tool_version`：激活工具版本表
- `activation_code_inventory`：激活码库存表
- `file_asset`：文件资源表

## 7.2 激活交付扩展表

### 7.2.1 激活工具主表 `activation_tool`

| 字段名 | 类型 | 说明 |
|---|---|---|
| tool_id | UUID | 主键 |
| merchant_id | UUID | 商家 ID |
| product_id | UUID | 商品 ID |
| tool_name | varchar(128) | 工具名称 |
| current_version_id | UUID | 当前生效版本 ID，可为空 |
| tool_status | varchar(32) | 工具启停状态 |
| audit_status | varchar(32) | 审核状态 |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

**状态定义：**

- `tool_status`：`ENABLED` / `DISABLED`
- `audit_status`：`PENDING` / `APPROVED` / `REJECTED`

**约束建议：**

- 主键：`pk_activation_tool(tool_id)`
- 外键：`merchant_id -> merchant.merchant_id`
- 外键：`product_id -> product.product_id`
- 外键：`current_version_id -> activation_tool_version.tool_version_id`
- 唯一约束：`uk_activation_tool_product_id(product_id)`

**索引建议：**

- `uk_activation_tool_product_id`
- `idx_activation_tool_merchant_status (merchant_id, tool_status)`
- `idx_activation_tool_current_version_id (current_version_id)`

---

### 7.2.2 激活工具版本表 `activation_tool_version`

| 字段名 | 类型 | 说明 |
|---|---|---|
| tool_version_id | UUID | 主键 |
| tool_id | UUID | 工具 ID |
| version_name | varchar(64) | 版本号 |
| file_id | UUID | 上传文件 ID |
| manifest_content | jsonb | 解析后的 manifest 内容 |
| runtime_type | varchar(32) | 运行类型 |
| runtime_os | varchar(32) | 目标运行系统 |
| runtime_arch | varchar(32) | 目标架构 |
| entrypoint | varchar(255) | 入口文件 |
| exec_command | text | 执行命令模板 |
| timeout_seconds | int | 超时时间（秒） |
| max_memory_mb | int | 最大内存（MB） |
| file_size_bytes | bigint | 文件大小（字节） |
| checksum_sha256 | varchar(64) | 文件 SHA-256 校验值 |
| scan_status | varchar(32) | 文件扫描状态 |
| scan_report | text | 文件扫描报告 |
| review_remark | text | 审核备注 |
| status | varchar(32) | 版本状态 |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

**补充说明：**

- `manifest_content` 建议采用 `jsonb`，便于后续结构化查询
- `runtime_type` 建议取值：`jar`、`shell`、`native_binary`
- `exec_command` 仅保存平台定义的命令模板，不直接拼接用户原始输入

**状态定义：**

- `scan_status`：`PENDING` / `SAFE` / `DANGEROUS` / `FAILED`
- `status`：`DRAFT` / `PENDING` / `ENABLED` / `DISABLED` / `REJECTED`

> 仅 `status = ENABLED` 的版本允许被设置为当前生效版本。

**约束建议：**

- 主键：`pk_activation_tool_version(tool_version_id)`
- 外键：`tool_id -> activation_tool.tool_id`
- 外键：`file_id -> file_asset.file_id`
- 唯一约束：`uk_tool_version_tool_id_version_name(tool_id, version_name)`

**索引建议：**

- `uk_tool_version_tool_id_version_name`
- `idx_tool_version_tool_id_status (tool_id, status)`
- `idx_tool_version_file_id (file_id)`

## 8. 激活交付能力设计

## 8.1 设计背景与目标

为支持商品多样化交付，系统扩展两类模式：

1. **激活码库存交付**：商家上传激活码库存，订单履约时分配激活码
2. **激活工具执行交付**：商家上传工具包，平台在隔离环境中执行工具完成交付

本次设计目标包括：

- 支持 `CODE_STOCK` 与 `TOOL_EXECUTION` 两种交付模式
- 支持激活工具版本化管理
- 支持文件元数据管理与对象存储
- 支持审核、扫描、启停与版本切换
- 支持高并发异步履约
- 为后续拆分 `activation-runner` 预留边界

---

## 8.2 设计原则

1. **数据库存储业务元数据，二进制文件存对象存储**
2. **激活工具采用版本化管理**
3. **工具执行与主业务服务解耦**
4. **重操作流程优先异步化**
5. **Redis 仅作为性能与协调组件，不作为最终一致性数据源**

---

## 8.3 对象存储与上传规范

### 8.3.1 存储方案

激活工具相关二进制内容不直接存储于 PostgreSQL，采用：

- **数据库**：保存文件元数据、校验值、路径与状态
- **对象存储**：保存文件本体

推荐方案：

- 开发环境：`MinIO`
- 生产环境：MinIO 集群或兼容 S3 的对象存储

### 8.3.2 对象路径规范

建议采用分层命名，例如：

```text
activation-tools/{merchantId}/{toolId}/{toolVersionId}/package.zip
```

或：

```text
activation-tools/{yyyy}/{MM}/{dd}/{uuid}.zip
```

要求：

- 不直接使用用户原始文件名作为对象路径
- 路径中应包含业务维度或随机因子
- 便于定位、排查与治理

### 8.3.3 上传方式

**初期方案：**

- 由 `actrade-api` 接收上传后转存对象存储

**推荐演进方案：**

- 使用预签名 URL 直传对象存储
  1. 客户端申请上传凭证
  2. 业务服务生成预签名地址
  3. 客户端直传对象存储
  4. 上传完成后业务服务登记 `file_asset`

### 8.3.4 工具包规范

推荐上传压缩包结构：

```text
tool-package.zip
├── manifest.json
├── bin/
│   └── xxx.jar / xxx.sh / xxx.exe
└── resources/
```

示例 `manifest.json`：

```json
{
  "toolName": "ExampleActivator",
  "version": "1.0.0",
  "runtimeType": "jar",
  "runtimeOs": "linux",
  "runtimeArch": "amd64",
  "entrypoint": "bin/app.jar",
  "args": ["--orderNo", "{orderNo}"],
  "timeoutSeconds": 60,
  "maxMemoryMb": 256
}
```

### 8.3.5 上传校验要求

至少包括：

- 扩展名白名单校验
- MIME 类型校验
- 文件大小限制校验
- 压缩包结构合法性校验
- 路径穿越风险校验
- 解压后文件数量与总体积限制校验
- SHA-256 校验值计算
- 恶意文件扫描
- Manifest 解析校验

---

## 8.4 服务拆分与基础设施职责

### 8.4.1 服务职责

#### `actrade-api`

负责：

- 商品管理
- 工具主记录与版本记录管理
- 文件元数据登记
- 工具审核流程
- 订单履约调度
- 激活码库存分配
- 与 Redis / RabbitMQ / 对象存储交互

#### `activation-runner`

负责：

- 拉取工具包
- 在隔离环境中执行工具
- 收集执行日志与结果
- 回传执行状态与执行产物
- 处理执行超时、失败与资源回收

### 8.4.2 基础设施职责

- **PostgreSQL**：保存业务主数据、工具元数据、文件元数据、库存数据与审核状态
- **Redis**：缓存热点库存、幂等控制、短期状态缓存、分布式锁、限流防重
- **RabbitMQ**：异步任务分发与结果回传
- **MinIO**：工具包与相关文件存储

### 8.4.3 推荐消息主题

- `file.scan.request`
- `file.scan.result`
- `tool.execution.submit`
- `tool.execution.result`
- `inventory.code.release`

---

## 8.5 并发设计

### 8.5.1 总体原则

- 同步链路尽量短
- 重操作异步化
- 热点数据缓存化
- 状态变更幂等化
- 关键库存以数据库原子更新为准

### 8.5.2 激活码库存策略

建议流程：

1. 支付成功后触发履约
2. 数据库原子更新一条 `AVAILABLE` 记录为 `LOCKED`
3. 绑定订单与订单项
4. 履约成功后改为 `SOLD`
5. 若失败或超时，异步回滚为 `AVAILABLE` 或置为 `VOID`

> Redis 可缓存可售数量，但最终库存分配结果以数据库事务为准。

### 8.5.3 激活工具执行策略

建议流程：

1. 订单支付成功
2. `actrade-api` 创建履约任务并投递 RabbitMQ
3. `activation-runner` 消费任务并执行
4. 执行结果通过消息或回调写回业务服务
5. 业务服务更新履约状态并通知用户

---

## 8.6 安全设计

### 8.6.1 基本原则

激活工具属于用户上传的可执行内容，存在注入、恶意程序、越权访问与资源滥用风险。  
平台需采用“独立服务 + 隔离环境 + 最小权限”的执行策略。

### 8.6.2 禁止事项

禁止：

- 在 `actrade-api` 进程内直接执行工具
- 在宿主机直接运行商家上传文件
- 以 root 权限运行用户上传内容
- 允许执行容器访问宿主机敏感目录
- 允许执行容器共享宿主机网络、PID 或 IPC 命名空间

### 8.6.3 执行隔离要求

`activation-runner` 执行时应满足：

- 使用独立运行容器或独立沙箱
- 默认非 root 运行
- 禁止特权模式
- 禁止权限提升
- 限制 CPU、内存、执行时长
- 默认禁止外部网络访问
- 使用临时工作目录
- 执行完成后销毁运行环境与临时文件

### 8.6.4 审核与扫描要求

- 文件安全扫描
- Manifest 结构检查
- 工具版本审核
- 启用前二次校验
- 风险版本可停用与回滚

如需增强恶意文件检测能力，后续可增加容器化扫描组件，例如 `actrade-clamav`。

## 8.7 典型业务流程

### 8.7.1 激活码交付流程

1. 商家创建 `delivery_mode = CODE_STOCK` 的商品
2. 商家批量导入激活码库存
3. 系统写入 `activation_code_inventory`
4. 用户下单并支付
5. 系统锁定一条可用库存
6. 履约成功后库存状态更新为 `SOLD`
7. 向用户展示脱敏码及必要信息

### 8.7.2 激活工具交付流程

1. 商家创建 `delivery_mode = TOOL_EXECUTION` 的商品
2. 商家上传工具包至对象存储
3. 系统创建 `file_asset`
4. 系统解析 manifest 并写入 `activation_tool_version`
5. 审核通过后切换 `activation_tool.current_version_id`
6. 用户下单并支付
7. `actrade-api` 投递执行任务至 RabbitMQ
8. `activation-runner` 拉取工具包并在隔离环境中执行
9. 执行结果回传主业务服务
10. 主业务服务完成履约状态更新
