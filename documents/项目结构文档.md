# ACTrade 项目结构与系统设计文档

## 1. 文档说明

本文档用于说明 ACTrade 项目的当前目录结构、后端分层设计、技术栈、核心数据模型，以及围绕“激活码 / 激活工具”交付能力的扩展设计，供团队内部评审、开发协作与后续归档使用。

---

## 2. 项目概览

ACTrade 是一个交易平台项目，当前以后端 API 工程为核心，基于 Spring Boot 3 + Java 21 开发。  
现阶段已完成用户、商家、商品、商品分类、商品详情等基础能力，并已规划激活码库存和激活工具执行两类交付模式。

### 2.1 模块说明

- `actrade-api`：后端 API 工程，承载当前主要业务实现，已包含用户、商家、商品、订单、支付、激活交付等核心模块
- `actrade-web`：前端工程目录，当前仍作为预留目录，尚未补充正式实现
- `documents`：项目文档目录，存放项目结构、计划、设计说明等文档
- `infra`：基础设施目录，当前包含本地依赖服务的 Docker Compose 配置

## 3. 后端架构设计

### 3.1 分层结构

后端代码采用**按业务模块组织、模块内分层**的方式，整体结构清晰，便于后续演进与拆分。

各层职责如下：

- `api`：控制器层，对外暴露 REST API
- `api/request`：请求 DTO
- `api/response`：响应 DTO
- `application`：应用服务层，负责业务流程编排
- `application/payment`：支付领域应用层，封装支付处理器、执行结果、工厂等
- `domain`：领域模型层，包含实体、枚举等核心对象
- `persistence`：持久化层，包含 Repository 与查询能力
- `infrastructure`：基础设施适配层，当前主要承载支付宝网关接入
- `common`：通用响应模型、公共异常
- `config`：项目配置
- `web`：Web 层统一异常处理

这种结构使系统具备以下特点：

1. **业务模块边界清晰**
2. **支付接入与主业务逻辑解耦**
3. **后续支持工具执行、消息队列、对象存储等能力扩展**
4. **便于后续拆分独立服务**

---

### 3.2 当前业务模块

当前后端已包含以下主要业务模块：

- `app_users`：用户模块
- `merchant`：商家模块
- `product`：商品与商品详情模块
- `order`：订单与支付模块
- `activation`：激活码库存、激活工具、工具版本、文件资源模块
- `common`：通用响应与异常模型
- `config`：配置模块
- `web`：统一异常处理模块

其中：

- `order` 模块已支持订单提交、订单查询、支付发起、支付回调处理
- `order.infrastructure.alipay` 已接入支付宝沙箱
- `activation` 模块已完成围绕“激活码库存交付 / 激活工具执行交付”的基础建模与接口组织

## 6. 核心数据模型设计

### 6.1 基础业务模型

当前系统核心基础模型包括：

- 用户 `app_users`
- 商家 `merchant`
- 商品分类 `product_category`
- 商品 `product`
- 商品详情 `product_detail`

这些表构成平台的基础商品管理能力。

---

### 6.2 订单与支付模型

当前系统已新增订单交易模型：

- 订单主表 `orders`
- 订单项表 `order_items`

其中：

- `orders` 用于存储订单主信息、支付状态、支付方式、第三方支付流水标识等
- `order_items` 用于存储订单中的商品快照与数量、单价、小计金额

当前支付相关数据未单独拆表，而是直接记录在 `orders` 中，主要字段包括：

- `payment_method`
- `payment_status`
- `payment_request_no`
- `channel_trade_no`
- `paid_at`

这种设计适合当前阶段的单支付主流程实现，结构简单，便于快速联调。  
后续若扩展退款、分账、支付历史、支付重试等复杂场景，可考虑拆分独立支付单表。

---

### 6.3 交付能力扩展模型

为支持不同交付模式，系统扩展以下模型：

- `activation_tool`：激活工具主表
- `activation_tool_version`：激活工具版本表
- `activation_code_inventory`：激活码库存表
- `file_asset`：文件资源表

这些模型用于支撑：

1. **激活码库存交付**
2. **激活工具执行交付**

- 订单支付状态与履约状态分离，便于后续扩展履约流程
- `seller_id` 允许为空，用于部分订单可能未指定卖家（如平台自营）
- `platform_income_amount` 和 `seller_income_amount` 在结算时计算，可暂时为空

**索引与约束：**

- 主键：`seller_settlements_pkey (settlement_id)`
- 唯一约束：
  - `uk_seller_settlements_settlement_no (settlement_no)`
  - `uk_seller_settlements_order_id (order_id)` —— 一个订单仅对应一条结算记录
- 索引：
  - `idx_seller_settlements_created_at (created_at)`
  - `idx_seller_settlements_merchant_id (merchant_id)`
  - `idx_seller_settlements_order_id (order_id)`
  - `idx_seller_settlements_seller_id (seller_id)`
  - `idx_seller_settlements_status (settlement_status)`

**检查约束：**

- `seller_settlements_settlement_status_check`：结算状态枚举
  - `UNSETTLED`、`PENDING`、`SETTLED`

**说明：**

- 每个订单最终生成一条卖家结算记录（一对一关系）
- `settlement_status` 表示结算流程状态：`UNSETTLED`（待结算）、`PENDING`（结算中）、`SETTLED`（已结算）
- `settled_at` 记录实际完成结算的时间

## 8. 激活交付能力设计

### 8.1 设计背景与目标

为支持商品多样化交付，系统目前围绕两类模式进行设计与实现：

1. **激活码库存交付**
2. **激活工具执行交付**

当前系统已具备订单与支付主流程，因此激活交付能力将以“**支付成功后触发履约**”为主线继续扩展。

本阶段设计目标包括：

- 支持 `CODE_STOCK` 与 `TOOL_EXECUTION` 两种交付模式
- 支持工具版本化管理
- 支持文件元数据管理
- 支持审核、扫描、启停与版本切换
- 支持订单支付成功后的履约衔接
- 为后续拆分 `activation-runner` 预留边界

---

### 8.2 设计原则

1. **数据库存储业务元数据，二进制文件存对象存储**
2. **支付与履约解耦**
3. **工具执行与主 API 服务解耦**
4. **重操作优先异步化**
5. **库存分配以数据库事务结果为准**
6. **状态流转必须保证幂等**
7. **Redis 仅作为协调与性能组件，不作为最终一致性数据源**

---

### 8.3 对象存储与上传规范

#### 8.3.1 存储方案

激活工具相关二进制内容不直接存储于 PostgreSQL，而采用：

- **数据库**：保存文件元数据、校验值、路径、扫描状态
- **对象存储**：保存文件本体

推荐方案：

- 开发环境：`MinIO`
- 生产环境：MinIO 集群或兼容 S3 的对象存储

#### 8.3.2 对象路径规范

建议采用分层命名，例如：

```text
activation-tools/{merchantId}/{toolId}/{toolVersionId}/package.zip
```

或：

```text
activation-tools/{yyyy}/{MM}/{dd}/{uuid}.zip
```

要求：

- 不直接使用用户原始文件名作为对象路径
- 路径中应包含业务维度或随机因子
- 便于治理、排查与归档

#### 8.3.3 上传方式

**初期方案：**

- 由 `actrade-api` 接收上传后转存对象存储

**推荐演进方案：**

- 使用预签名 URL 直传对象存储：
  1. 客户端申请上传凭证
  2. 业务服务生成预签名地址
  3. 客户端直传对象存储
  4. 业务服务登记 `file_asset`

#### 8.3.4 工具包规范

推荐上传压缩包结构：

```text
tool-package.zip
├── manifest.json
├── bin/
│   └── xxx.jar / xxx.sh / xxx.exe
└── resources/
```

#### 8.3.5 上传校验要求

至少包括：

- 扩展名白名单校验
- MIME 类型校验
- 文件大小限制校验
- 压缩包结构合法性校验
- 路径穿越风险校验
- 解压后文件数量与总体积限制校验
- SHA-256 校验值计算
- 恶意文件扫描
- Manifest 解析校验

---

### 8.4 服务拆分与基础设施职责

#### 8.4.1 服务职责

##### `actrade-api`

负责：

- 用户、商家、商品管理
- 订单创建与订单查询
- 支付发起与支付回调处理
- 激活工具与版本管理
- 文件元数据登记
- 支付成功后的履约调度
- 激活码库存分配
- 与 Redis / RabbitMQ / MinIO / 支付网关集成

##### `activation-runner`

负责：

- 拉取工具包
- 在隔离环境中执行工具
- 收集执行日志与执行结果
- 回传履约结果
- 处理超时、失败与资源回收

---

### 8.5 并发设计

#### 8.5.1 总体原则

- 同步链路尽量短
- 重操作异步化
- 支付回调处理幂等化
- 库存分配原子化
- 履约结果更新幂等化

#### 8.5.2 支付处理原则

当前支付能力已在订单表上完成主状态建模，建议遵循以下原则：

1. 发起支付时生成或复用 `payment_request_no`
2. 支付处理中将订单置为 `PAYING`
3. 异步通知以 `payment_request_no` 或渠道交易标识做幂等校验
4. 成功通知只允许将订单推进到 `PAID`
5. 已处理成功的通知再次到达时应直接幂等返回

#### 8.5.3 激活码库存策略

建议流程：

1. 订单支付成功
2. 系统原子锁定一条 `AVAILABLE` 库存
3. 绑定订单与订单项
4. 履约成功后更新为 `SOLD`
5. 失败或超时则异步释放

#### 8.5.4 激活工具执行策略

建议流程：

1. 订单支付成功
2. `actrade-api` 创建履约任务并投递消息
3. `activation-runner` 执行工具
4. 结果回传主服务
5. 主服务更新履约状态并通知用户

---

### 8.6 安全设计

#### 8.6.1 基本原则

激活工具属于用户上传的可执行内容，存在注入、恶意程序、越权访问与资源滥用风险。平台必须采用：

- 独立执行服务
- 隔离运行环境
- 最小权限原则
- 严格的审核与扫描机制

#### 8.6.2 禁止事项

禁止：

- 在 `actrade-api` 进程内直接执行工具
- 在宿主机直接执行商家上传文件
- 以 root 权限运行用户上传内容
- 允许访问宿主机敏感目录
- 允许共享宿主机网络、PID 或 IPC 命名空间

#### 8.6.3 执行隔离要求

`activation-runner` 执行时应满足：

- 使用独立容器或隔离沙箱
- 默认非 root 运行
- 禁止特权模式
- 禁止权限提升
- 限制 CPU、内存、执行时长
- 默认禁止外部网络访问
- 使用临时工作目录
- 执行完成后销毁运行环境

---

### 8.7 典型业务流程

#### 8.7.1 订单支付流程

1. 用户选择商品并提交订单
2. 系统创建 `orders` 与 `order_items`
3. 用户发起支付
4. 系统根据支付方式生成支付请求
5. 若为支付宝，则生成页面支付表单并跳转收银台
6. 支付宝异步通知回调主系统
7. 系统验签并更新订单为已支付
8. 支付成功后进入履约流程

#### 8.7.2 激活码交付流程

1. 商家创建 `CODE_STOCK` 商品
2. 商家导入激活码库存
3. 用户下单并支付
4. 支付成功后系统锁定库存
5. 库存交付成功后更新为 `SOLD`
6. 向用户展示交付结果

#### 8.7.3 激活工具交付流程

1. 商家创建 `TOOL_EXECUTION` 商品
2. 商家上传工具包并登记文件元数据
3. 系统创建版本信息
4. 审核通过后启用版本
5. 用户下单并支付
6. 支付成功后投递执行任务
7. `activation-runner` 拉取工具并执行
8. 回传执行结果
9. 主业务服务更新履约状态
